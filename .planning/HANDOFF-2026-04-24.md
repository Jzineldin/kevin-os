## The in-flight blocker

**Goal:** dashboard Capture box → APIGW → dashboard-api Lambda → EventBridge → triage → voice-capture → Notion Command Center row.

**Status:** Works until triage. Triage receives the event (the `kind:'text'` fix in `8f8832d` got the EventBridge rule to match) but triage immediately ZodError-rejects on schema validation.

**Concrete error from triage Lambda logs** (capture_id `01KQ08WRFP4HWGSWZVB8ZR9T96`):
```
ZodError: [
  { "code": "invalid_literal", "expected": "telegram", "path": ["channel"] },
  { "code": "invalid_type", "expected": "object", "received": "undefined", "path": ["sender"] },
  { "code": "invalid_type", "expected": "object", "received": "undefined", "path": ["telegram"] }
]
```

**Root cause:** `packages/contracts/src/events.ts:33-47` — `CaptureReceivedTextSchema` is hard-coded to `channel: z.literal('telegram')` with required `sender` + `telegram` objects. Dashboard-api emits `channel: 'dashboard'` without those fields.

**The fix:** Widen the schema. Recommended shape:
```typescript
export const CaptureReceivedTextSchema = z.object({
  capture_id: z.string().regex(UlidRegex),
  channel: z.enum(['telegram', 'dashboard']),
  kind: z.literal('text'),
  text: z.string().min(1).max(8000),
  received_at: z.string().datetime(),
  // Telegram-only fields become optional:
  sender: z.object({
    id: z.number().int(),
    display: z.string().optional(),
  }).optional(),
  telegram: z.object({
    chat_id: z.number().int(),
    message_id: z.number().int(),
  }).optional(),
  source: z.string().optional(),  // dashboard sends this
});
```

Then **rebuild + redeploy the triage Lambda** (I did it via direct `aws lambda update-function-code` earlier — same path should work for triage). Path: `services/triage/src/handler.ts`, function name `KosAgents-TriageAgentA93D4BB1-f7aRCapBeNp9`.

## Major architectural discoveries today (document in follow-ups)

### 1. AWS Organization SCP blocks Lambda Function URL AuthType=NONE

Kevin's AWS account is inside AWS Organization `o-g7pp3kd04f` (root `051542583152`). Some SCP denies `AuthType=NONE` Lambda Function URLs. Symptom: `403 AccessDeniedException` with "Function URL authorization issues" message even when resource policy explicitly allows `Principal: "*"`.

**Workaround used:** put API Gateway HTTP API in front instead. API Gateway HTTP API IS allowed — the Telegram webhook (`TelegramWebhookApi`, id `b81fkhaybe`) uses it successfully.

**New dashboard HTTP API:** id `v1k7d48lbk`, endpoint `https://v1k7d48lbk.execute-api.eu-north-1.amazonaws.com/`. Routes:
- `ANY /{proxy+}` → dashboard-api Lambda
- `GET/POST /relay/{proxy+}` → relay-proxy Lambda

**Undocumented sub-mystery:** even under AuthType=AWS_IAM, the `kos-dashboard-caller` IAM user gets 403 on the Function URL despite matching identity + resource policies. Admin role works. STS session tokens derived from caller keys also fail. Root cause never identified. API Gateway sidesteps this entirely.

### 2. Bearer auth replaces SigV4 on dashboard-api + relay-proxy

Code changes in commit `1af37a6`:
- `services/dashboard-api/src/index.ts` — added `verifyBearer` check before `route(event)`
- `services/dashboard-listen-relay/src/proxy.ts` — same pattern
- `apps/dashboard/src/lib/dashboard-api.ts` — Vercel client now sends `Authorization: Bearer <KOS_DASHBOARD_BEARER_TOKEN>` instead of SigV4 via aws4fetch

Both Lambdas have `KOS_DASHBOARD_BEARER_TOKEN` env var set from `kos/dashboard-bearer-token` Secret. Security posture: same shared secret that gates the /login page.

**Cleanup task:** remove the now-unused `AWS_ACCESS_KEY_ID_DASHBOARD` + `AWS_SECRET_ACCESS_KEY_DASHBOARD` env vars from Vercel. Remove aws4fetch from `apps/dashboard/package.json` if nothing else uses it. Remove the caller user resource-policy statements on the Lambdas (SigV4 path dead).

### 3. Next.js 16 + Serwist incompatibility

Bumping Next 15.5.4 → 16.2.4 broke Serwist's service worker emission. `/sw.js` now returns 404. PWA installability lost. Options: downgrade Next + find a patched 15.x; or wait for Serwist Next-16 support; or switch to workbox-webpack-plugin directly.

### 4. Locked Decision #3 drift, formally recorded

Claude Agent SDK was abandoned in Wave 5 (can't run in Lambda — spawns `claude` CLI subprocess stripped by esbuild). Replacement: direct `AnthropicBedrock` SDK calls. Documented in `PROJECT.md` Key Decisions table (commit `88b8e0d`). Phase 6 AGT-04 redesigned accordingly as explicit `@kos/context-loader::loadContext` helper rather than SDK pre-call hook.
