| `KosSafety-PushTelegram…` | `Dynamic require of "events" is not supported` | pg@8.13.1/lib/client.js |

CloudWatch metrics confirm the live impact: during the
`verify-resolver-three-stage.mjs` run at 21:08–21:09 Stockholm,
`KosAgents-EntityResolver*` registered **42 invocations / 42 errors** (100%
error rate) — every one of the 20 published `entity.mention.detected` events
plus EventBridge rule retries failed at INIT before any business logic ran.

#### Root cause

esbuild (via the KosLambda construct's NodejsFunction) is bundling the
service code as ESM (`format: 'esm'`, output `index.mjs`) but the bundled
dependencies (`@opentelemetry/core`, `grammy`, `pg`) contain `require(...)`
calls that esbuild's ESM output rewrites into a `__require()` shim. That
shim then fails at runtime because Lambda's Node 22 ESM loader does not
expose CommonJS-compatible `require` for the bundled file
(`Dynamic require of "X" is not supported` is the exact error string esbuild
emits in this case).

#### Fix paths (NOT to be applied in this plan — Plan 02-09 / 02-10 territory)

1. **Cleanest:** flip KosLambda's NodejsFunction `format` from `esm` to
   `cjs`, and rename the output to `index.js`. Lambda's Node 22 runtime
   defaults to CJS unless `package.json` says otherwise; this aligns with
   the bundled dependencies.
2. **Banner shim:** keep ESM output but inject
   `--banner:js="import { createRequire } from 'module'; const require =
   createRequire(import.meta.url);"` so the `__require()` shim resolves.
   This is the documented esbuild-ESM-on-Lambda workaround and is the
   smallest diff.
3. **Externalise the offenders:** mark `pg`, `@opentelemetry/*`, `grammy`
   as Lambda layers / external modules — heaviest lift, not recommended.

**Recommended action:** add a Plan 02-12 (or rework Plan 02-09's tracing
helper) to apply fix path #2 across every KosLambda invocation, then redeploy
all 7 stacks and re-run this gate.
