    **`packages/cdk/lib/stacks/integrations-discord-schedule.ts`**:
    ```typescript
    import type { Construct } from 'constructs';
    import { Duration } from 'aws-cdk-lib';
    import { CfnSchedule } from 'aws-cdk-lib/aws-scheduler';
    import { Role, ServicePrincipal, PolicyStatement, Effect } from 'aws-cdk-lib/aws-iam';
    import { StringParameter } from 'aws-cdk-lib/aws-ssm';

    export interface WireDiscordScheduleProps {
      kevinOwnerId: string;
    }

    export function wireDiscordSchedule(scope: Construct, props: WireDiscordScheduleProps) {
      // Phase 10 owns the Lambda; this phase just points a Scheduler rule at an ARN pulled from SSM.
      // If Phase 10 hasn't landed yet, operator can seed the SSM param with a no-op ARN (or leave it
      // dangling — Scheduler will fail-quiet; CloudWatch alarm picks it up).
      const arnParam = StringParameter.fromStringParameterName(
        scope, 'DiscordBrainDumpLambdaArn',
        '/kos/discord/brain-dump-lambda-arn',
      );

      const schedulerRole = new Role(scope, 'DiscordScheduleRole', {
        assumedBy: new ServicePrincipal('scheduler.amazonaws.com'),
      });
      schedulerRole.addToPolicy(new PolicyStatement({
        effect: Effect.ALLOW,
        actions: ['lambda:InvokeFunction'],
        resources: [arnParam.stringValue],  // the SSM value at synth-time; Phase 10 may update the param + redeploy
      }));

      new CfnSchedule(scope, 'DiscordBrainDumpSchedule', {
        name: 'kos-discord-poll',
        scheduleExpression: 'cron(0/5 * * * ? *)',   // every 5 min, UTC (Discord messages timezone-neutral)
        scheduleExpressionTimezone: 'UTC',
        flexibleTimeWindow: { mode: 'OFF' },
        target: {
          arn: arnParam.stringValue,
          roleArn: schedulerRole.roleArn,
          input: JSON.stringify({
            // Scheduler input consumed by Phase 10 Lambda. Contract documented in 05-06-DISCORD-CONTRACT.md.
            channel: 'brain-dump',
            owner_id: props.kevinOwnerId,
            trigger_source: 'kos-discord-poll-scheduler',
          }),
          retryPolicy: { maximumRetryAttempts: 2, maximumEventAgeInSeconds: 300 },
        },
        state: 'ENABLED',
      });
    }
    ```

    Wire into `IntegrationsStack`. Note the SSM parameter:
    - Phase 5 expects it pre-seeded (operator `aws ssm put-parameter --name /kos/discord/brain-dump-lambda-arn --type String --value <arn>`).
    - If Phase 10 hasn't shipped, seed with a no-op Lambda (e.g., AWS Organization's default `arn:aws:lambda:eu-north-1:<acct>:function:no-op`) or a throwaway Lambda that returns 200 for any event.

    Write the cross-phase contract:

    ```markdown
    # CAP-10 Discord Brain-Dump Contract

    **Phase 5 Plan 05-06** (this file) owns the EventBridge Scheduler.
    **Phase 10 Plan 10-04** owns the `discord-brain-dump-listener` Lambda handler.

    ## Scheduler → Lambda Input

    The Scheduler fires every 5 min UTC and invokes the Lambda with:
    ```json
    {
      "channel": "brain-dump",
      "owner_id": "<kevin-uuid>",
      "trigger_source": "kos-discord-poll-scheduler"
    }
    ```

    ## Lambda → kos.capture Output

    The Lambda MUST emit one `capture.received` per new Discord message:
    ```json
    {
      "capture_id": "<26-char-ULID>",
      "channel": "discord",
      "kind": "discord_text",
      "channel_id": "<discord-channel-snowflake>",
      "message_id": "<discord-message-snowflake>",
      "author": { "id": "<snowflake>", "display": "<name>" },
      "body": "<text>",
      "sent_at": "<ISO datetime>",
      "received_at": "<ISO datetime>"
    }
    ```

    Schema: `@kos/contracts/CaptureReceivedDiscordTextSchema` (shipped in Plan 05-00).

    ## Idempotency

    `capture_id` must be deterministic from `(channel_id, message_id)` (sha256 → 26-char Crockford base32).
    Duplicate processing on Scheduler retries or Phase 10 Lambda retries produces the same capture_id; downstream dedupes.

    ## Cursor

    Phase 10 Lambda maintains a cursor `discord_last_seen_message_id` per channel in RDS (table + column name at Plan 10-04 discretion). Scheduler input doesn't carry the cursor — it's Lambda-local state.

    ## Rate-limit budget

    Discord API: 50 requests/second per bot token (global), 10/s per channel read. Phase 10 Lambda fetches `GET /channels/<channel_id>/messages?after=<cursor>` once per invocation (default 50 messages max). At 12 invocations/hour = well under budget.

    ## Graceful degradation (D-22)

    - Bot token revoked → 401 from Discord → Lambda writes `system_alerts` row + updates `sync_status.channel='discord'`.
    - Channel archived → 404 from Discord → same treatment.
    - No Telegram fire-alarm. Dashboard surfaces. Next daily brief mentions.

    ## Deploy order resilience

    - If Phase 10 lands BEFORE Phase 5: Phase 10 writes Lambda → seeds SSM param. Phase 5 reads SSM + creates Scheduler.
    - If Phase 5 lands FIRST: Phase 5 Scheduler points at placeholder no-op ARN (operator-seeded). Phase 10 updates SSM param + may require Scheduler redeploy to pick new ARN (CDK stack change triggers update).
    - Either order is safe.
    ```
