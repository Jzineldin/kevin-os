# Pattern: how to target an EventBridge bus from Scheduler
@packages/cdk/lib/stacks/integrations-notion.ts

<interfaces>
<!-- EventBridge Scheduler targets -->
EventBridge Scheduler can target ANY AWS service via the universal target API. For posting to an EventBridge bus:
- target.arn = the bus ARN (NOT a Lambda ARN)
- target.roleArn = an IAM role with events:PutEvents on the bus
- target.input = JSON string (the single event body that will be PutEvents'd)
- Scheduler effectively wraps target.input in a PutEventsRequest with Source/DetailType pulled from … **wait — we need to verify this**.

After verification (RESEARCH §2 + AWS Scheduler docs):
EventBridge Scheduler universal targets for EventBridge requires `events:PutEvents` permission. However, the TARGET format for `scheduler → events:PutEvents` specifically needs:
- `target.arn` = `arn:aws:scheduler:::aws-sdk:eventbridge:putEvents`  (universal target template)
- `target.roleArn` = role with events:PutEvents
- `target.input` = JSON matching `PutEventsRequest`:
  ```json
  { "Entries": [{ "EventBusName": "kos.system", "Source": "kos.system", "DetailType": "scan_emails_now", "Detail": "{...}" }] }
  ```

Simpler alternative: target a Lambda that PutEvents on the system bus. But Phase 4 already has a Lambda (email-triage) that SUBSCRIBES to the rule; creating a mini-publisher Lambda just for scheduling is overhead. The universal-target path is the right shape.

Actually SIMPLEST path: Scheduler target = email-triage Lambda directly + pass the event envelope. That works because the Lambda already dispatches on `event['detail-type'] === 'scan_emails_now'`. The scheduler's target.input becomes the Lambda event directly — no bus hop needed.

**LOCKED APPROACH** (per D-16 revised): EventBridge Scheduler target = **email-triage Lambda directly**. target.input = `{ "detail-type": "scan_emails_now", "source": "kos.system", "detail": { "capture_id": "<placeholder>", "requested_at": "<<aws.scheduler.scheduled-time>>", "requested_by": "scheduler" } }`.

This avoids the EventBridge bus hop AND matches Phase 4's dispatch semantics exactly. Phase 4's rule-based path (capture.received → email-triage) stays for push invocations; the scheduler-based path (scan_emails_now → email-triage) is the second trigger. Both paths coexist unchanged.

Caveat: Phase 7 must import the email-triage Lambda reference. IntegrationsStack either:
- Holds a reference to email-triage Lambda from its `wireEmailAgents` call (Plan 04-04 Task 4) and passes it into `wireLifecycleAutomation`, OR
- wireLifecycleAutomation accepts a `emailTriageFn` prop that IntegrationsStack wires.

Decision: add `emailTriageFn: KosLambda` to `WireLifecycleAutomationProps`; IntegrationsStack threads the return value of `wireEmailAgents` into the call.
