    **`scripts/power-down-hetzner.sh`** (bash, +x bit, operator-run):
    - `set -euo pipefail`.
    - Env: `KOS_HETZNER_SERVER_ID` (default: operator-prompt), `KOS_EVENT_LOG_LAMBDA_ARN` (for audit write).
    - Pre-flight:
      - `hcloud server describe $SERVER_ID` — asserts `status == 'running'`.
      - `hcloud context active` — asserts correct project.
      - Interactive confirmation prompt: "This will power off 98.91.6.66. Type POWEROFF to proceed:" — abort on mismatch.
    - Step 1: `hcloud server create-image --type snapshot --description "kos-vps-final-$(date -I)" --label kos-retention=30days $SERVER_ID` — capture `image-id` from stdout.
    - Step 2: Verify snapshot is `available`: poll `hcloud image describe $IMAGE_ID` until `status=available` (timeout 20 min).
    - Step 3: Write event_log row via AWS Lambda invoke (`aws lambda invoke --function-name $KOS_EVENT_LOG_LAMBDA_ARN --payload '{"kind":"hetzner-snapshot-created","details":{"image_id":"'$IMAGE_ID'","labels":["kos-retention=30days"]}}' /tmp/resp.json`).
    - Step 4: `hcloud server poweroff $SERVER_ID` — poll until `status=off` (timeout 5 min).
    - Step 5: Write event_log row `kind=vps-powered-down` with `details.server_id` + `details.powered_down_at`.
    - Step 6: Print operator-friendly summary + point to verify-hetzner-dead.mjs.
    - Exit 0 on success.

    **`scripts/verify-hetzner-dead.mjs`** (node, +x bit):
    - Target: 98.91.6.66 (const from env `KOS_HETZNER_TARGET_IP`).
    - Ports: [22, 80, 443, 5678].
    - For each port, `net.connect({ port, host, timeout: 5000 })` with listeners on `connect`, `timeout`, `error`.
    - Expected outcome per port: `ECONNREFUSED` OR `ETIMEDOUT` OR `EHOSTUNREACH`. FAIL if any port establishes a connection (`connect` event fires).
    - Also calls Hetzner billing API (if `HETZNER_API_TOKEN` set): `GET https://api.hetzner.cloud/v1/servers/$SERVER_ID/metrics?type=cpu&start=<now-24h>&end=<now>`; assert CPU usage averages 0 (powered off).
    - Output: JSON summary `{ ip, ports: [{ port, state }], billing_zero: bool }`.
    - Exit 0 on all-closed, 1 on any-open.

    **`scripts/verify-telegram-webhook-persistence.mjs`** (node, +x bit):
    - Env: `TELEGRAM_BOT_TOKEN` (from Secrets Manager; operator exports).
    - `WEBHOOK_URL` from env (or from `scripts/.kos-config.json`).
    - Flow:
      1. Call `POST https://api.telegram.org/bot${TOKEN}/setWebhook?url=${WEBHOOK_URL}`. Assert `ok:true`.
      2. Sleep 60 seconds via `await new Promise(r => setTimeout(r, 60000))`.
      3. Call `GET https://api.telegram.org/bot${TOKEN}/getWebhookInfo`. Parse `result.url`.
      4. If `result.url === WEBHOOK_URL`: log `[OK] Telegram webhook persisted after 60s — MIG-02 + INF-11 closure resolves telegram-webhook-auto-clear.md M1`. Emit event_log `kind=telegram-webhook-persistence-resolved`. Exit 0.
      5. If `result.url === ''`: log `[ESCALATE] Telegram webhook cleared within 60s — rogue caller remains. Run remediation per .planning/debug/telegram-webhook-auto-clear.md (Test 3 dev machine hunt OR rotate bot token).` Exit 2 (distinct from non-zero "error").
    - Structural assertions (grep targets from validation matrix row 7.4):
      - `getWebhookInfo` literal.
      - `setTimeout.*60000` literal.

    Include explicit guard for quiet hours: the test should NOT run between 20:00 and 08:00 Stockholm since that's when the real webhook receives no traffic and validation is ambiguous.
